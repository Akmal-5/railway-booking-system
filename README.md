# Railway Booking System

Backend-сервис для бронирования билетов на поезд, написанный на **FastAPI** с асинхронным доступом к **PostgreSQL** через **SQLAlchemy 2.0 (async)**. Сервис позволяет посмотреть список поездов, маршрут конкретного поезда, свободные места между двумя станциями и купить билет.

## Как это работает

1. При запуске приложения (`main.py`) срабатывает событие `startup`:
   - вызывается `create_tables()` — SQLAlchemy создаёт в базе все таблицы, описанные в моделях (`trains`, `stations`, `seats`, `train_routes`, `tickets`), если их ещё нет;
   - дальше идёт **сидинг (seed) тестовых данных**: создаётся поезд "Сапсан", создаются 7 станций маршрута Москва → Тверь → Вышний Волочек → Бологое → Чудово → Санкт-Петербург → Великий Новгород, для поезда создаются 12 мест (`Seats`, номера 1–12) и привязывается маршрут (`TrainRoute`) с порядковым номером каждой станции.
   - Все функции сидинга в `data_manipulation.py` сначала проверяют, не созданы ли уже такие записи, поэтому повторный запуск не создаёт дубликаты.
2. После старта приложение регистрирует три роутера (`trains_router`, `seats_router`, `tickets_routers`), каждый из которых отвечает за свою группу эндпоинтов.
3. Каждый запрос получает свою асинхронную сессию БД через dependency `create_session` (FastAPI `Depends`), которая открывает сессию на время обработки запроса и закрывает её после ответа.
4. Вся логика работы с БД (запросы, проверки, бизнес-правила) находится в `app/db/data_manipulation.py` — роутеры лишь принимают параметры запроса и передают их в эти функции.

## Модель данных

```
Trains (поезда)                Stations (станции)
├── id                          ├── id
└── name                        └── name (уникальное)

Seats (места)                  TrainRoute (маршрут поезда)
├── id                          ├── id
├── train_id  -> Trains.id      ├── train_id    -> Trains.id
└── number                      ├── stations_id -> Stations.id
                                 └── order (порядковый номер станции в маршруте)

Tickets (билеты)
├── id
├── seat_id          -> Seats.id
├── from_station_id  -> Stations.id
├── to_station_id    -> Stations.id
└── passenger_name
```

Идея модели: у каждого поезда есть набор мест (`Seats`) и маршрут (`TrainRoute`) — упорядоченный список станций. Билет (`Tickets`) не привязан к конкретному отрезку напрямую через поезд, а связывает место с парой станций отправления/назначения. Занятость места на конкретном участке маршрута вычисляется на лету путём сравнения порядковых номеров (`order`) станций.

## API: список маршрутов

| Метод | Путь | Тег | Описание |
|---|---|---|---|
| GET | `/trains/` | Trains | Список всех поездов |
| GET | `/trains/{train_id}/route` | Trains | Маршрут поезда: список станций в порядке следования |
| GET | `/trains/{train_id}/seats` | Места | Список свободных мест на участке между двумя станциями |
| POST | `/trains/{train_id}/tickets` | Tickets | Покупка билета (бронирование места) |

### GET `/trains/`
Возвращает список всех поездов из таблицы `trains` (на старте там уже будет один поезд — "Сапсан").

Пример ответа:
```json
[
  { "id": 1, "name": "Сапсан" }
]
```

### GET `/trains/{train_id}/route`
Принимает `train_id` (целое число, путь). Возвращает список станций маршрута в порядке следования, с их порядковым номером.

Если поезда с таким `id` нет — ответ `404 Not Found` ("Поезд не найден").

Пример ответа:
```json
[
  { "order": 1, "station_id": 1, "station_name": "Москва" },
  { "order": 2, "station_id": 2, "station_name": "Тверь" }
]
```

### GET `/trains/{train_id}/seats`
Принимает `train_id` (путь) и query-параметры `from_station_id`, `to_station_id`. Возвращает список мест, которые свободны на запрашиваемом участке маршрута.

Логика проверки занятости: место считается занятым, если на него уже есть билет, чей отрезок маршрута (от/до станции) пересекается с запрашиваемым отрезком (по порядковым номерам станций, а не по их id).

Если одна из станций не входит в маршрут поезда — ответ `404 Not Found` ("Станция не найдена в маршруте").

### POST `/trains/{train_id}/tickets`
Покупка билета. Принимает `train_id` (путь) и тело запроса:
```json
{
  "seat_id": 3,
  "from_station_id": 1,
  "to_station_id": 6,
  "passenger_name": "Иван Иванов"
}
```
Перед созданием билета сервер повторно вычисляет список свободных мест и проверяет, что `seat_id` входит в него. Если место уже занято на этом отрезке — ответ `400 Bad Request` ("Это место уже занято"). При успехе создаётся запись в таблице `tickets` и возвращается созданный билет.

## Логика покупки билета

Файл `app/db/data_manipulation.py`, функция `buy_ticket`:

1. Вызывает `get_available_seats`, чтобы получить актуальный список свободных мест для пары станций.
2. Проверяет, что переданное в запросе `seat_id` есть среди свободных.
3. Если нет — `400 Bad Request`.
4. Если есть — создаёт запись `Tickets` и сохраняет её в БД.

Функция `get_available_seats` сравнивает не сами `station_id`, а их порядковые номера (`order`) в маршруте поезда — это позволяет корректно определять пересечение отрезков пути (например, билет "Москва → Санкт-Петербург" блокирует место для попытки купить "Тверь → Чудово", хотя станции отправления/назначения разные).

## Запуск проекта

### Вариант 1 — Docker Compose (рекомендуется)

```bash
docker-compose up --build
```

Поднимутся два контейнера:
- `db` — PostgreSQL 16 (порт `5432`);
- `web` — FastAPI-приложение (порт `8000`), которое ждёт, пока база станет готова (healthcheck), затем запускается.

После запуска документация Swagger доступна по адресу: `http://localhost:8000/docs`

### Вариант 2 — локально, без Docker

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env             # и подставить свою строку подключения к БД
uvicorn main:app --reload
```

Для этого варианта нужна заранее запущенная PostgreSQL и переменная `DATABASE_URL` в `.env`.

## Переменные окружения

Файл `.env` (создаётся на основе `.env.example`):

```env
DATABASE_URL=postgresql+asyncpg://<user>:<password>@<host>:<port>/<dbname>
```

В `docker-compose.yml` для контейнера `web` эта переменная уже прописана и указывает на сервис `db` внутри docker-сети:
```
DATABASE_URL=postgresql+asyncpg://postgres:postgres@db:5432/railway_booking_system
```
